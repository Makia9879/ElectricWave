## Question

这个通知应用的 MVP 要服务哪些具体用户场景？需要明确最先支持的通知来源、触发方式、用户看到通知后的期望动作，以及哪些场景明确不进入第一版。

## Resolution

MVP 采用静态三段架构：

```text
webhook -> 服务器 -> 手机
```

第一版服务的核心场景是：外部系统通过服务器暴露的 webhook 入口发送通知请求，服务器完成访问控制与接收端匹配，然后把通知转发到已配置的手机 App。手机 App 负责把服务端转发的消息发布为本机通知，并尽量适配 HyperOS 灵动岛/系统通知展示能力。

前置条件：

- 服务器程序必须配置 webhook 访问令牌，外部调用方只有持有该令牌才能发送通知。
- 服务器程序必须配置接收端白名单，只有被允许的手机接收端可以接收转发。
- 手机 App 必须配置通知发布点，用于接收服务器转发的通知。
- 手机 App 必须配置接收端身份令牌，服务器用它识别和授权具体接收端。

第一版进入范围：

- 单向通知链路：外部系统调用 webhook，服务器转发到手机，手机展示通知。
- 访问令牌控制 webhook 调用权限。
- 接收端白名单控制服务器允许转发到哪些手机端。
- 手机端身份令牌用于绑定或识别接收端。

第一版不进入范围：

- 手机端主动订阅任意外部 webhook。
- 无服务器的纯本机或局域网直连模式。
- 多用户账号体系、商业化权限体系、复杂组织管理。
- 通知后的双向操作闭环；MVP 先以展示通知为主。
