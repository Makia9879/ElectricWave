# 定义连接状态实时 UX 模型

Status: closed

## Question

App 应如何实时展示连接状态？需要定义用户可理解的状态集合、状态来源、刷新频率、最近心跳/最近错误/积压数量展示、一键重连入口、权限或系统限制的诊断提示，以及哪些状态不应该误导用户以为通知一定可达。

## Resolution

连接状态 UI 采用“可达性 + 展示能力 + 积压”三层模型，而不是单一在线/离线。核心原则：UI 可以说“连接正常、待补发 3 条”，但不能说“通知一定可达”或“已展示”，因为 App 只能确认 SSE 事件已 ack，无法确认系统通知最终出现在用户眼前。

### 首页状态集合

首页只展示 6 个用户可理解状态：

- `接收中`：profile enabled，前台服务运行，SSE 已连接，最近心跳正常，通知权限和目标渠道允许。
- `正在重连`：profile enabled，前台服务运行，但 SSE 断开或心跳超时，当前处于指数退避或立即重连中。
- `有待补发`：服务端返回 receiver backlog 大于 0，或本地知道存在未 ack 事件；可与“正在重连”并存，但首页优先显示待补发数量。
- `需要授权`：通知权限、渠道、前台服务通知、自启动/后台运行或电池策略存在阻断项。
- `已暂停`：用户关闭 profile 或点击停止接收；此状态不自动重连。
- `不可用`：server endpoint、证书、receiver token、receiver revoked/disabled、服务端 401/403/404 等配置级错误，需要用户修改配置或重新绑定。

### 诊断明细

状态详情页展示机器可读但不吓人的字段：

- 服务端地址、receiver_id、profile enabled。
- 前台服务：运行中/未运行/被用户停止/未知。
- SSE：connected/reconnecting/disconnected/auth_failed/server_unavailable。
- 最近心跳时间、最近连接时间、最近断开时间、下一次重连时间。
- 服务端积压数量、最老积压时间、最近 ack event id。
- 通知展示条件：通知权限、默认渠道、紧急渠道、锁屏可见性、勿扰影响只做诊断，不承诺绕过。
- 系统可靠性条件：开机自启动、后台运行、电池策略。Android 标准无法稳定读取所有 HyperOS 开关时，标记为“需要用户确认”，并提供设置入口。
- 最近错误：HTTP 状态、网络错误分类、TLS 错误、token 失效、backlog gap、TTL 过期数量。

### 刷新与数据来源

- 本地连接状态由前台服务通过应用内状态存储实时更新；首页可 1 秒内反映连接/断开/重连中。
- 心跳时间使用服务端 SSE heartbeat 或控制事件更新时间；若超过 2 个 heartbeat 周期未更新，进入“正在重连”。
- 积压数量来自服务端 receiver status 或 SSE 控制事件；没有服务端结果时显示“未知”，不要猜。
- 权限和渠道状态在 App 启动、回到前台、用户打开诊断页和系统设置返回时刷新。
- 不在后台做高频 UI 轮询；后台只由前台服务维护连接状态，UI 恢复时读取最近快照。

### 一键重连入口

首页在以下状态显示主按钮：

- `正在重连`：按钮文案“立即重连”，点击后重置退避并发起一次连接。
- `有待补发`：按钮文案“重连并补发”，点击后尝试连接并请求从最近 ack/cursor 补发。
- `需要授权`：按钮文案随第一阻断项变化，例如“开启通知权限”“允许自启动”“调整电池策略”；如果无法直达系统设置，打开说明页。
- `不可用`：按钮文案“检查配置”或“重新绑定”，不盲目重连。
- `已暂停`：按钮文案“开启接收”，用户点击后才启动前台服务。

### 不应出现的误导文案

- 不使用“100% 可达”“后台永久在线”“已送达手机”“已展示通知”。
- `acked` 只能展示为“App 已收到”，不能展示为“用户已看到”。
- 自启动/省电设置未确认时，不显示“开机自动恢复已开启”。
- 强行停止、系统 Stop、HyperOS 清理后，只能在下次打开 App 时提示“上次可能被系统或用户停止”，不能承诺自动检测实时发生。
