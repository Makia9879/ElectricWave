# 可靠连接与重连体验 Wayfinder Map

Labels: wayfinder:map
Status: closed
Assignee: Codex Agent Team

## Destination

明确自建 SSE 通知链路在“断线、重连、消息积压、手机端自启动与连接状态 UX”上的可实现规格：服务端是否以及如何保留待投递消息，Android/HyperOS 端何时自动重连、何时必须用户手动介入，App 如何实时呈现连接状态并支持一键重连。

## Notes

- 这是本仓库第二个话题 issue，独立于已关闭的 `0001-map-hyperos-dynamic-island-notification-app.md`，但承接其个人 MVP 结论：默认下行为 Android 前台服务维持的认证 HTTP SSE。
- 先做规划与决策，不直接实现服务端或 Android 端代码。
- 不承诺绕过 Android/HyperOS 的强行停止、省电、自启动限制；需要把系统限制明确暴露给产品与验收标准。
- 需要外部知识时优先查官方 Android 文档、小米/HyperOS 可获得资料，以及可验证的同类实现证据。
- 需要用户偏好判断时使用 grilling，一次只问一个问题。
- 新 wayfinder 规划按话题文件夹保存；本话题目录内的票据从 `0001` 开始使用局部下标，不沿用其他话题的全局下标。

## Decisions so far

- [决定服务端消息积压与重连补发语义](0002-server-backlog-and-replay-semantics.md) — 服务端采用短期、有界、按 receiver 积压的 backlog，并通过 SSE `id`、`Last-Event-ID` 和 `X-Receiver-Ack` 实现重连补发；ack 只表示 App 已收到，不承诺系统通知已展示。
- [调研 Android/HyperOS 自启动与后台恢复边界](0003-android-hyperos-autostart-and-background-recovery.md) — 自动恢复只在用户已启用接收、前台服务可运行、系统未拦截自启动/后台运行且网络可用时成立；强行停止、系统 Stop、HyperOS 清理和严格省电不承诺自动恢复。
- [定义连接状态实时 UX 模型](0004-connection-status-ui-model.md) — UI 拆成连接、展示能力和积压三层，首页只展示接收中、正在重连、有待补发、需要授权、已暂停、不可用，避免“100% 可达/已展示”等误导文案。
- [决定一键重连与应用活动状态自动重连策略](0005-manual-and-lifecycle-reconnect-policy.md) — 统一 guard 后再重连；一键重连、回前台、网络恢复可立即尝试，普通错误指数退避，认证/配置/用户停止类错误停止自动重连。
- [汇总可靠连接与重连体验规格](0006-reliable-reconnect-spec.md) — 已形成可交给实现 agent 的服务端 backlog、Android 前台服务、生命周期重连、状态 UI、诊断和验收矩阵规格。

## Out of scope

- 直接实现完整服务端队列、Android 前台服务或 UI。
- 多设备同步、账号体系、跨平台客户端、商业化推送通道。
- 承诺应用被用户强行停止、系统禁止自启动、通知权限关闭或渠道关闭后仍能自动恢复通知展示。
